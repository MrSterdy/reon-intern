# amoCRM Contacts And Leads Integration

## Описание проекта

Backend-приложение на NestJS для интеграции с amoCRM. Приложение должно обрабатывать установку и удаление интеграции, хранить токены аккаунтов, синхронизировать кастомные поля, устанавливать вебхуки и обрабатывать карточки "Контакт" и "Сделка".

## Стек технологий

- Backend: NestJS
- База данных: MongoDB + Mongoose или PostgreSQL + TypeORM
- Контейнеризация: Docker, docker-compose
- Интеграция: amoCRM API v4

## Техническое задание

### 1. Модуль интеграции с amoCRM

Необходимо реализовать отдельный модуль для работы с amoCRM API.

Функциональность:

- принимать хук установки интеграции из amoCRM;
- обменивать authorization code на `accessToken` и `refreshToken`;
- сохранять аккаунт в базе данных;
- если аккаунт уже существует, обновлять его данные;
- при хуке удаления интеграции:
  - установить `accessToken = null`;
  - установить `refreshToken = null`;
  - установить `isInstalled = false`;
- реализовать Cron-задачу, которая каждые 12 часов обновляет токены для всех установленных аккаунтов.

Модель аккаунта:

```ts
accountId: string | number; // id аккаунта в amoCRM, уникальное поле
subdomain: string;
accessToken: string | null;
refreshToken: string | null;
isInstalled: boolean;
```

Критерии приемки:

- аккаунт создается при установке интеграции;
- повторная установка обновляет существующий аккаунт;
- удаление интеграции не удаляет аккаунт из базы, а сбрасывает токены и `isInstalled`;
- Cron-задача обновляет токены только для установленных аккаунтов;
- работа с amoCRM API вынесена в отдельный модуль/сервис.

### 2. Синхронизация кастомных полей amoCRM

После установки интеграции необходимо получить кастомные поля из amoCRM и проверить наличие обязательных полей.

Если нужного поля нет в amoCRM, его необходимо создать. После проверки/создания все обязательные поля нужно сохранить в базе данных с привязкой к аккаунту.

Кастомные поля контакта:

| Название поля | Тип amoCRM |
| --- | --- |
| Дата рождения | Дата |
| Возраст | Число |
| Лазерное омоложение лица | Число |
| Ультразвуковой лифтинг | Число |
| Лазерное удаление сосудов | Число |
| Коррекция мимических морщин | Число |
| Лазерная эпиляция | Число |

Кастомные поля сделки:

| Название поля | Тип amoCRM |
| --- | --- |
| Услуги | Мультисписок |
| Бюджет | Стандартное поле amoCRM, число |

Модель кастомного поля:

```ts
fieldType: string; // тип поля в amoCRM
fieldId: string | number; // id поля в amoCRM
fieldName: string; // уникальное название поля из констант проекта
accountId: string; // id аккаунта в базе проекта
```

Требования:

- названия обязательных полей должны быть вынесены в константы;
- проверка существования поля выполняется по названию и типу;
- у каждого аккаунта должны сохраняться собственные `fieldId`;
- поле `fieldName` используется для поиска нужного поля в базе;
- поле "Бюджет" является стандартным полем amoCRM и не создается как кастомное.

Критерии приемки:

- все обязательные поля контакта проверяются и создаются при отсутствии;
- поле "Услуги" для сделки проверяется и создается при отсутствии;
- все нужные `fieldId` сохраняются в базе с привязкой к аккаунту;
- повторная синхронизация не создает дубликаты.

### 3. Проверка и установка вебхуков amoCRM

Необходимо через amoCRM API проверить наличие вебхуков на события:

- контакт добавлен;
- контакт изменен;
- сделка добавлена;
- сделка изменена.

Если вебхуки отсутствуют, их необходимо установить.

Критерии приемки:

- приложение получает список установленных вебхуков аккаунта;
- отсутствующие вебхуки создаются;
- повторная установка интеграции не создает дубликаты вебхуков;
- URL вебхуков берется из env-конфигурации.

### 4. Модуль работы с контактами

Необходимо реализовать модуль обработки контактов.

Функциональность:

- принимать вебхуки amoCRM на добавление и изменение контакта;
- получать данные контакта через amoCRM API;
- проверять заполненность поля "Дата рождения";
- если "Дата рождения" не заполнена, ничего не делать;
- если "Дата рождения" заполнена:
  - вычислить возраст;
  - обновить поле "Возраст" в карточке контакта.

Особое требование:

- после обновления контакта amoCRM повторно отправит вебхук изменения;
- необходимо предусмотреть защиту от лишней повторной обработки, например не обновлять контакт, если рассчитанный возраст уже совпадает со значением поля.

Критерии приемки:

- возраст корректно рассчитывается по дате рождения;
- поле "Возраст" обновляется только при необходимости;
- при отсутствии даты рождения запрос на обновление контакта не выполняется;
- повторный вебхук после обновления не приводит к бесконечному циклу.

### 5. Модуль работы со сделками

Необходимо реализовать модуль обработки сделок.

Функциональность:

- принимать вебхуки amoCRM на добавление и изменение сделки;
- получать данные сделки через amoCRM API;
- проверять выбранные значения в кастомном поле "Услуги";
- если услуги не выбраны, ничего не делать;
- если услуги выбраны:
  - получить основной контакт, привязанный к сделке;
  - проверить у контакта заполненность полей выбранных услуг;
  - проверить наличие возраста контакта;
  - при необходимости создавать или обновлять задачи в карточке сделки;
  - при успешной проверке рассчитать стоимость услуг и обновить поле "Бюджет".

#### Сценарий 1. Не заполнены поля выбранных услуг

Если у контакта не заполнены поля услуг, выбранных в сделке, необходимо создать задачу в карточке сделки.

Текст задачи:

```text
У контакта не заполнены поля услуг: [перечислить выбранные услуги у которых не заполнены поля]
```

Требования:

- тип задачи "ошибка" берется из env;
- дедлайн задачи: 24 часа;
- если такая задача уже есть, новую не создавать;
- если текст существующей задачи отличается, обновить его.

#### Сценарий 2. Не заполнен возраст контакта

Если поле "Возраст" не заполнено:

- проверить, можно ли рассчитать возраст по полю "Дата рождения";
- если можно, вызвать алгоритм обработки возраста контакта;
- если нельзя, создать задачу в карточке сделки.

Текст задачи:

```text
Возраст контакта неизвестен
```

Требования:

- тип задачи "ошибка" берется из env;
- дедлайн задачи: 24 часа;
- если такая задача уже есть, ничего не делать.

#### Сценарий 3. Все необходимые поля заполнены

Если выбранные услуги заполнены и возраст контакта известен:

- рассчитать стоимость услуг;
- обновить стандартное поле сделки "Бюджет";
- создать или обновить задачу в карточке сделки.

Текст задачи:

```text
Проверить стоимость услуг для [Название контакта], возраст: [Возраст контакта]
```

Требования:

- тип задачи "проверить" берется из env;
- дедлайн задачи: 24 часа;
- если такая задача уже есть, обновить ее при необходимости.

Общие требования к задачам:

- все задачи создаются только для карточки "Сделка";
- ID типов задач должны быть описаны в env;
- для тестирования можно использовать стандартные amoCRM task type id `0` и `1` либо созданные вручную типы задач.

Критерии приемки:

- сделки без выбранных услуг не обрабатываются;
- при отсутствии основного контакта приложение не падает;
- задачи-дубликаты не создаются;
- существующие задачи обновляются только при необходимости;
- бюджет сделки обновляется после успешной проверки данных;
- логика расчета стоимости вынесена в отдельный сервис/метод и покрыта тестами.

### 6. Docker и env-конфигурация

Необходимо создать папку:

```text
docker-compose
```

Внутри нее добавить файл для запуска приложения и базы данных.

Требования:

- docker-compose должен поднимать backend-приложение;
- docker-compose должен поднимать базу данных;
- для базы должен быть настроен volume, чтобы данные сохранялись между перезапусками;
- необходимо добавить `.env.example` для запуска приложения и базы;
- env example должен содержать все обязательные переменные.

Пример обязательных env-переменных:

```env
APP_PORT=

DATABASE_HOST=
DATABASE_PORT=
DATABASE_NAME=
DATABASE_USER=
DATABASE_PASSWORD=
DATABASE_URL=

AMO_CLIENT_ID=
AMO_CLIENT_SECRET=
AMO_REDIRECT_URI=
AMO_INTEGRATION_BASE_URL=

AMO_WEBHOOK_CONTACT_CREATED_URL=
AMO_WEBHOOK_CONTACT_UPDATED_URL=
AMO_WEBHOOK_LEAD_CREATED_URL=
AMO_WEBHOOK_LEAD_UPDATED_URL=

AMO_ERROR_TASK_TYPE_ID=
AMO_CHECK_TASK_TYPE_ID=
```

Критерии приемки:

- приложение запускается через docker-compose;
- база данных поднимается вместе с приложением;
- данные базы не теряются после перезапуска контейнеров;
- `.env.example` содержит все переменные, необходимые для локального запуска.

## Правила выполнения

Разработка выполняется по правилам backend-разработки REON:

- соблюдать раздел "NEW Правила оформления кода Backend";
- соблюдать раздел "NEW Структура Backend проекта";
- соблюдать правила работы с GitHub.

GitHub flow:

1. Создать репозиторий с веткой `main`.
2. От `main` создать ветку `development`.
3. В `development` сделать init commit с чистым NestJS-проектом.
4. Каждый пункт ТЗ выполняется отдельным Pull Request.
5. Для каждого пункта создается отдельная ветка от `development`.
6. После проверки PR мержится в `development`.
7. Следующая задача начинается от актуальной `development`.

Разбиение по PR:

- PR 1: amoCRM OAuth, аккаунты, установка/удаление интеграции, Cron обновления токенов.
- PR 2: синхронизация кастомных полей и сохранение `fieldId` в базе.
- PR 3: проверка и установка вебхуков amoCRM.
- PR 4: модуль контактов и расчет возраста.
- PR 5: модуль сделок, задачи, проверка услуг, расчет бюджета.
- PR 6: docker-compose, env example, документация по запуску.

Общие критерии готовности:

- код соответствует структуре NestJS-проекта;
- все env-переменные валидируются при старте приложения;
- бизнес-логика покрыта unit-тестами там, где это уместно;
- интеграционные точки amoCRM вынесены в отдельные сервисы;
- README содержит инструкцию по запуску и настройке интеграции;
- приложение не создает дубликаты полей, вебхуков и задач при повторной обработке.
